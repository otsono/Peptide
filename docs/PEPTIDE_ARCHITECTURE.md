# Peptide architecture — what lives where, and why

Peptide drives a running Fraymakers engine for testing converted characters (and,
increasingly, for live mod development). This doc records the **layering
decisions** so the system stays maintainable and a future GUI / `.hl` migration
can build on a stable base. The guiding rule:

> **Push logic to the highest, most readable layer it can live in.**
> Hand-written HashLink opcodes are the layer of last resort.

## The four layers (lowest → highest)

| Layer | Where | What lives here | Why |
|---|---|---|---|
| 1. Hand-written bytecode | `src/main.rs` `connect_edit` | The per-frame dispatch skeleton injected into `fraymakers.Main.update`: socket connect/auth, the single-byte command read, and the handler bodies. | HashLink has no plugin hook; we must splice ops into the engine's own bytecode. This is irreducible — but it should stay **thin**. |
| 2. Rust-generated bytecode | `src/main.rs` + `src/asm.rs` | Op sequences emitted from Rust data tables (the `move` jump table from `commands::MOVES`; the `physics` field reads). The `Asm` helper resolves jump offsets at build time. | Generating ops from a table is far safer than hand-writing each: add a move = one table row, not 5 hand-placed opcodes with hand-computed offsets. This is the **preferred** place for new engine-side logic that can be expressed as "read fields / call methods / format output". |
| 3. Client (Rust, no engine) | `src/main.rs` + `src/interpreter.rs` | The human-facing command vocabulary, name→wire translation, reply glossing, help. Zero engine bytecode. | Renaming/adding a friendly command, prettifying output, validating args — none of this should touch the engine. A future GUI talks to this layer. |
| 4. Shell orchestration | `tools/run.sh`, `tools/runseq.sh`, `tools/tests/batch_spawn_test.sh` | Boot lifecycle, multi-command sequencing, batch sweeps, cleanup. | Process/lifecycle concerns, not protocol. |

## The wire protocol (the seam between layer 1 and layer 3)

Deliberately minimal: **one byte selects a handler**; only `s` (spawn) and the
arg-bearing commands drain a trailing line. The engine never parses words.

- The human types `spawn sandbag` / `move special_neutral` / `physics`.
- `commands::translate()` (layer 3) turns that into the wire line `s sandbag` /
  `m N` / `v`.
- The engine dispatch (layer 1/2) matches the first byte and runs the handler.

Keeping the protocol byte-thin is *why* the friendly-rename cost zero engine
bytecode, and why `move <name>` only needed a Rust-generated jump table rather
than string parsing in opcodes.

## Decision log

- **Friendly command names live client-side (not engine-side).** Parsing command
  *words* in HashLink opcodes would be a large, fragile hand-bytecode effort.
  Putting the vocabulary in `interpreter.rs` (shared by patcher + bridge) made it a
  data-only concern. *(landed)*
- **`move <name>` selects CState via a Rust-generated jump table.** The bridge
  sends `m <selector>` where `selector = 'A' + ordinal` into `commands::MOVES`;
  the engine compares the byte against generated `JEq` arms. CState fields are
  resolved **by name** at patch time (drift-robust). No integer parsing, no
  `alloc_array` — only opcodes already proven in this build. *(landed)*
- **`physics` formats Floats with the engine's own `Std.string`.** Avoids writing
  a float→string routine in bytecode. **Gotcha learned:** HL derives a `Field`'s
  byte offset from the destination reg's *static type*, so each component
  (Body/Physics/Damage) needs a correctly-typed reg — reusing one Body-typed reg
  to read Physics fields returns uninitialized-memory garbage. *(landed)*
- **`Asm` (`src/asm.rs`) is mandatory for any new handler block.** It allocates
  registers and resolves label jumps at build time, turning a class of runtime
  engine crashes (mis-jumps, clobbered regs) into Rust compile/patch-time errors.
  New handlers should be self-contained `Asm` blocks that fall off their end into
  the next check. *(convention)*

## Where new features should go (the `.hl` direction — greenlit)

For features that exceed "read a few fields / call a method / format a line",
**do not keep hand-writing opcodes.** The sanctioned escalation, in order:

1. **Express it as a Rust-generated `Asm` block** if it's still fundamentally
   field-reads + method-calls driven by a data table (layer 2). Most readback
   commands fit here.
2. **Write it in Haxe, compile offline to `.hl`, load via a minimal bytecode
   bootstrap** (the greenlit migration) when the logic is real control flow /
   data structures / anything you'd hate to express in opcodes. The injected
   layer-1 skeleton shrinks to: connect socket + load the `.hl` module + forward
   commands to it. All feature logic lives in readable Haxe.
3. The bootstrap stays tiny and stable; everything new accretes in the Haxe layer.

This keeps the fragile hand-bytecode surface from growing, which is the stability
win the project is optimizing for. See `docs/PEPTIDE_FUTURE.md` for the feature
roadmap that will exercise this path.

**`.hl` feasibility caveat (must spike before committing).** HashLink bytecode is
a *monolithic* file; there is (as far as known) no runtime "load another `.hl`
module" facility. So "load via bootstrap" most likely means **merging** a Haxe-
compiled `.hl` into the engine bytecode *at patch time* — appending its functions/
types/strings and remapping every findex / type index / string index so the two
modules' tables don't collide, then wiring the injected dispatch skeleton to call
the merged-in entry points. `hlbc` (already a dependency) gives read/write access
to all those tables, so it's mechanically possible, but the index-remapping is
non-trivial and needs a dedicated feasibility spike (start with merging a
one-function `.hl` that returns a constant and calling it from the dispatch)
before any feature depends on it. Until that spike succeeds, complex features stay
on the Rust-generated-`Asm` path (layer 2) or are deferred.

## IP boundary

Peptide contains **no** Fraymakers code, bytecode, strings, or assets. It reads
the user's *local* engine bytecode at runtime, writes a patched *copy*, and
speaks a loopback TCP protocol. Method/field/type names appear only as
interoperability facts in our own words. See `NOTICE.md`.
